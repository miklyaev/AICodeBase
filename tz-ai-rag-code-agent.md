# Техническое задание: AI-агент для поиска релевантных мест в кодовой базе с использованием RAG

## 1. Назначение документа

Документ предназначен для передачи модели gpt-5.3-codex в качестве подробного технического задания на разработку веб-приложения.

Цель приложения: дать пользователю локальный веб-интерфейс, в котором он выбирает папку проекта на своем компьютере, приложение индексирует кодовую базу через RAG, а затем по текстовым запросам пользователя находит и показывает наиболее релевантные участки кода.

## 2. Краткое описание продукта

Необходимо разработать локальное веб-приложение с двумя основными частями:

1. Frontend: React/Vite приложение в стиле Windows 11.
2. Backend: локальный Node.js/NestJS сервер, который имеет доступ к файловой системе пользователя, сканирует выбранную папку проекта, строит локальный RAG-индекс и обрабатывает запросы поиска.

Пользовательский сценарий:

1. Пользователь запускает локальный backend и frontend.
2. В левой колонке интерфейса нажимает кнопку выбора папки проекта.
3. Backend открывает системный диалог выбора директории или принимает путь, выбранный через поддерживаемый механизм.
4. Backend сканирует проект, исключает нерелевантные и опасные файлы, разбивает код на чанки, получает embeddings через OpenRouter и сохраняет индекс локально в LanceDB/SQLite.
5. Пользователь вводит вопрос в чат.
6. Frontend отправляет сообщение на backend.
7. Backend выполняет retrieval по индексу, формирует контекст, отправляет запрос через OpenRouter к модели gpt-5.3-codex.
8. Модель возвращает объяснение и список найденных мест в коде.
9. Справа отображается найденный код с путями файлов, номерами строк, подсветкой синтаксиса и краткими пояснениями.

## 3. Выбранная архитектура

### 3.1. Компоненты

1. Frontend:
   - React.
   - Vite.
   - TypeScript.
   - CSS Modules, SCSS или vanilla CSS с дизайн-токенами Windows 11.
   - Monaco Editor или CodeMirror для отображения кода справа.
   - Клиентский state management: Zustand или React Context.

2. Backend:
   - Node.js.
   - NestJS.
   - TypeScript.
   - REST API для базового управления проектами, индексом и чатом.
   - WebSocket или Server-Sent Events для прогресса индексации и потоковой генерации ответа.

3. RAG-хранилище:
   - LanceDB для векторного поиска.
   - SQLite для метаданных проекта, файлов, чанков, истории чата, настроек и состояния индексации.

4. LLM и embeddings:
   - OpenRouter API.
   - Chat-модель: gpt-5.3-codex.
   - Embedding-модель: задается конфигурацией OpenRouter. Если конкретная embedding-модель недоступна через OpenRouter, предусмотреть конфигурируемый адаптер embeddings и понятную ошибку настройки.

### 3.2. Обоснование выбора

1. React/Vite обеспечивает быстрый frontend, удобную разработку и простую сборку.
2. NestJS подходит для структурированного backend с модулями, сервисами, dependency injection и типизацией.
3. LanceDB хорошо подходит для локального векторного поиска и хранения embeddings.
4. SQLite удобен как локальная база метаданных без отдельного сервера.
5. OpenRouter позволяет централизованно подключать gpt-5.3-codex и embedding-модели через один API key.
6. Разделение frontend/backend необходимо, потому что браузер не должен напрямую иметь полный доступ к локальной файловой системе.

## 4. Основные функциональные требования

### 4.1. Выбор папки проекта

1. В левой колонке сверху должна быть кнопка Выбрать папку проекта.
2. После выбора папки frontend должен показать:
   - название проекта;
   - абсолютный или сокращенный путь;
   - статус индексации;
   - количество найденных файлов;
   - количество проиндексированных чанков.
3. Backend должен проверять, что выбранный путь существует и является директорией.
4. Backend должен запрещать индексировать системные и опасные директории по умолчанию, если они не разрешены явно.
5. Повторный выбор другой папки должен сбрасывать активный индекс или создавать новый project profile.

### 4.2. Ввод OpenRouter API key

1. В интерфейсе должна быть отдельная секция настройки API key.
2. Поле ввода API key должно вести себя как пароль:
   - вводимые символы скрыты;
   - значение не отображается в открытом виде;
   - рядом допускается кнопка временного показа, но по умолчанию ключ всегда скрыт;
   - ключ не должен попадать в историю чата, логи frontend, URL, query string или сообщения об ошибках.
3. Ключ должен храниться одним из способов:
   - по умолчанию только в памяти процесса backend до завершения сессии;
   - опционально в локальном защищенном хранилище, если пользователь явно включит Запомнить ключ.
4. Если ключ не указан, приложение должно блокировать индексацию через внешние embeddings и чат-запросы с понятным уведомлением.

### 4.4. Индексация и RAG

Backend должен реализовать pipeline:

1. Project scan:
   - получить список файлов;
   - применить ignore rules;
   - вычислить hash содержимого каждого файла;
   - определить измененные, новые и удаленные файлы.

2. Parsing and chunking:
   - читать файл как текст;
   - определить язык по расширению;
   - разбить файл на чанки с учетом структуры кода;
   - сохранять для каждого чанка путь файла, диапазон строк, язык, имя ближайшей функции или класса при возможности;
   - использовать overlap между чанками для сохранения контекста.

3. Embeddings:
   - отправлять чанки в embedding API через OpenRouter;
   - батчировать запросы;
   - учитывать rate limits;
   - повторять временно неуспешные запросы с backoff;
   - сохранять embedding и metadata локально.

4. Vector search:
   - по пользовательскому вопросу получить embedding запроса;
   - найти top-K релевантных чанков в LanceDB;
   - дополнительно применить keyword search по SQLite FTS или аналогичному механизму;
   - объединить результаты vector search и keyword search;
   - применить reranking эвристиками: совпадение имени файла, пути, символов, языка, импортов, экспортов, тестов.

5. Context assembly:
   - собрать ограниченный контекст для LLM;
   - включить только релевантные чанки;
   - добавить метаданные: путь файла, строки, язык, краткое описание;
   - не отправлять файлы целиком без необходимости;
   - при нехватке контекста вернуть пользователю уточняющий вопрос или список возможных мест.

6. Answer generation:
   - отправить в OpenRouter запрос к gpt-5.3-codex;
   - модель должна вернуть структурированный ответ: краткое объяснение, список найденных файлов и диапазонов строк, уверенность, кодовые фрагменты.

### 4.5. Чат

1. Чат расположен в левой колонке.
2. Пользователь вводит запросы на естественном языке.
3. Примеры запросов:
   - Где реализована авторизация пользователя?
   - Найди компонент, который отвечает за модальное окно настроек.
   - Где происходит запрос к API создания заказа?
   - Покажи место, где валидируется email.
4. Чат должен поддерживать историю сообщений в рамках текущей сессии.
5. Каждое сообщение ассистента должно иметь ссылки на найденные кодовые фрагменты.
6. При клике по ссылке справа должен открываться соответствующий файл и диапазон строк.
7. Если проект еще не выбран или индекс не готов, чат должен показывать понятное состояние и не отправлять запросы.

### 4.6. Отображение найденного кода

Правая колонка занимает 60 процентов ширины.

Требования:

1. Показывать список найденных результатов сверху или в боковой панели правой колонки.
2. Для каждого результата отображать:
   - путь файла;
   - диапазон строк;
   - язык;
   - релевантность;
   - краткое объяснение, почему этот фрагмент подходит.
3. Основная область должна показывать код с подсветкой синтаксиса.
4. Должны подсвечиваться найденные строки.
5. Должна быть возможность переключаться между найденными фрагментами.
6. Желательно добавить кнопку копирования пути и фрагмента.
7. Если файл изменился после индексации, интерфейс должен показать предупреждение Индекс устарел.

## 6. Backend API

### 6.1. Общие требования

1. Все API доступны только на localhost по умолчанию.
2. CORS разрешен только для frontend origin.
3. Ответы должны быть типизированы.
4. Ошибки должны возвращаться в едином формате:
   - code;
   - message;
   - details;
   - requestId.

### 6.2. Рекомендуемые REST endpoints

1. POST /api/settings/openrouter-key
   - принимает API key;
   - валидирует формат;
   - сохраняет в памяти сессии или защищенном локальном хранилище по флагу remember.

2. GET /api/settings/status
   - возвращает, задан ли ключ;
   - не возвращает сам ключ.

3. POST /api/projects/select
   - запускает выбор или регистрацию папки проекта;
   - принимает путь, если системный диалог недоступен;
   - возвращает projectId, name, path, status.

4. POST /api/projects/index
   - запускает индексацию projectId.

5. GET /api/projects/:projectId/status
   - возвращает состояние сканирования и индексации.

6. POST /api/chat/message
   - принимает projectId, message, conversationId;
   - выполняет retrieval и LLM generation;
   - возвращает ответ и найденные code references.

7. GET /api/files/snippet
   - принимает projectId, filePath, startLine, endLine;
   - возвращает текст фрагмента с контекстными строками.

8. GET /api/search
   - выполняет только retrieval без генерации ответа;
   - полезно для отладки и быстрых поисковых запросов.

## 8. OpenRouter интеграция

### 8.1. Требования

1. Все запросы к OpenRouter идут только с backend.
2. Frontend не должен напрямую обращаться к OpenRouter.
3. API key не должен попадать в browser storage без явного решения пользователя.
4. Для chat completion использовать модель gpt-5.3-codex.
5. Для embeddings использовать конфигурируемую модель OpenRouter embeddings.
6. Добавить таймауты, retry, rate limit handling и понятные ошибки.

### 8.2. System prompt для chat-модели

Модель должна отвечать как агент поиска по коду.

Требования к ответу:

1. Не выдумывать файлы и строки.
2. Использовать только найденный RAG-контекст.
3. Если уверенность низкая, явно сообщить об этом.
4. Возвращать ссылки на конкретные файлы и строки.
5. Давать краткое объяснение, почему найденный фрагмент релевантен.
6. Не предлагать правки кода, если пользователь просит только найти место.
7. Если пользователь просит объяснить место, дать объяснение на основе кода.

## 9. Безопасность и приватность

1. Кодовая база пользователя остается на локальном компьютере.
2. Во внешние сервисы отправляются только выбранные чанки или embeddings-запросы, необходимые для RAG, если embeddings выполняются через OpenRouter.
3. Пользователь должен быть предупрежден, что при использовании OpenRouter часть кода может отправляться во внешний сервис для embeddings и генерации ответа.
4. Добавить настройку exclude secrets.
5. Никогда не индексировать реальные .env файлы по умолчанию.
6. Никогда не логировать API key.
7. Маскировать ключ в UI.
8. Backend слушает только 127.0.0.1 по умолчанию.
9. Запретить произвольное чтение файлов вне выбранного project root.
10. Нормализовать и проверять пути, чтобы избежать path traversal.
11. Добавить лимиты размера файла, количества файлов, общего размера проекта и размера запроса.
12. Добавить clear index action для удаления локальных данных проекта.

## 10. Нефункциональные требования

### 10.1. Производительность

1. Индексация должна быть инкрементальной.
2. Неизмененные файлы не должны переиндексироваться.
3. Большие файлы должны обрабатываться потоково или пропускаться по лимиту.
4. UI не должен блокироваться во время индексации.
5. Пользователь должен видеть прогресс.

### 10.2. Надежность

1. Ошибка индексации одного файла не должна останавливать весь процесс.
2. При сетевой ошибке OpenRouter показывать понятное сообщение и возможность повторить.
3. При перезапуске backend приложение должно восстанавливать список ранее выбранных проектов и состояние индекса, если данные сохранены.

### 10.3. Доступность

1. Все интерактивные элементы доступны с клавиатуры.
2. Поля имеют label.
3. Контрастность соответствует базовым требованиям WCAG.
4. Для прогресса индексации должны быть текстовые статусы, не только цвет.

4. В левой колонке сверху есть кнопка выбора папки проекта.
5. Пользователь может ввести OpenRouter API key в скрытое поле, где символы не видны.
6. Backend не возвращает API key ни в одном ответе.
7. После выбора проекта backend сканирует файлы и строит локальный индекс.
8. Пользователь видит прогресс индексации.
9. После индексации пользователь может задать вопрос в чат.

11. RAG retrieval возвращает релевантные чанки из проекта.
12. Правая колонка отображает найденный код с путями файлов и номерами строк.
13. Клик по результату в чате открывает соответствующий код справа.
14. Секретные файлы не индексируются по умолчанию.
15. При отсутствии API key или проекта UI показывает понятное состояние.

17. Индекс можно очистить.

## 14. Тестовые сценарии

### 14.1. Выбор проекта

1. Запустить frontend и backend.
2. Нажать Выбрать папку проекта.
3. Выбрать тестовый проект.
4. Проверить, что UI показывает имя проекта и путь.

### 14.2. API key

2. Убедиться, что символы скрыты.
3. Сохранить ключ в сессии.
4. Проверить status endpoint: он показывает только факт наличия ключа.
5. Проверить, что ключ не появляется в логах.

### 14.3. Индексация

1. Запустить индексацию.
2. Проверить прогресс.
3. Проверить, что node_modules и .git пропущены.
4. Проверить, что .env не индексируется.
5. Проверить, что LanceDB и SQLite содержат записи чанков.

### 14.4. Поиск места в коде

1. Задать вопрос: Где реализован роут логина?
2. Проверить, что система возвращает файлы и строки.
3. Проверить, что справа открывается код.
4. Проверить, что найденные строки подсвечены.

### 14.5. Ошибки

1. Удалить API key и отправить сообщение.
2. Проверить понятную ошибку.
3. Остановить интернет и повторить запрос.
4. Проверить retry или понятное сообщение о сетевой ошибке.

## 15. Ограничения и допущения

1. Приложение работает локально.
2. Backend имеет доступ к файловой системе только в рамках выбранного project root.
3. Выбор папки в обычном браузере напрямую невозможен для backend без посредника, поэтому требуется один из механизмов:
   - frontend получает путь через поддерживаемый локальный picker;
   - backend открывает системный диалог через Node.js библиотеку;
   - пользователь вручную вводит путь как fallback.
4. Основной целевой сценарий: запуск на машине разработчика.
5. Внешний OpenRouter используется для chat generation и embeddings, поэтому пользователь должен понимать, что релевантные фрагменты кода могут отправляться во внешний API.

## 17. Формат результата для chat endpoint

Backend должен возвращать структурированный объект со следующей логикой полей:

1. answer: краткий текст ответа ассистента.
2. confidence: high, medium или low.
3. references: список найденных мест.
4. references.filePath: относительный путь к файлу.
5. references.startLine: начальная строка.
6. references.endLine: конечная строка.
7. references.language: язык файла.
8. references.reason: почему фрагмент релевантен.
9. references.snippet: короткий фрагмент кода.
10. conversationId: идентификатор диалога.

## 19. Главный результат MVP

Должно получиться локальное веб-приложение, в котором пользователь выбирает папку проекта, вводит скрытый OpenRouter API key, ждет завершения RAG-индексации и затем через чат спрашивает, где в кодовой базе находится нужная логика. Приложение должно показать ответ в чате и открыть найденные участки кода справа с файлами, строками и подсветкой.
